# Фаза 2 — Меню, Покупки, Инвентарь

## 1. Цель

Замкнуть полный workflow ШефДома:
**Рецепт → Меню → Список покупок → Покупка → Инвентарь.**

Бекендовая часть end-to-end сценария:
1. Пользователь добавляет рецепт (Фаза 1, готово).
2. Пользователь создаёт меню и кладёт туда рецепт с количеством порций.
3. Пользователь генерирует список покупок из меню.
4. Пользователь отмечает позицию купленной.
5. Купленная позиция автоматически попадает в инвентарь
   (кладовка / холодильник / морозилка).

В Фазе 1 шаги 2–5 отсутствовали. Симптом, видимый из UI:
«Отмечаю товар купленным — не появляется в кладовке/холодильнике».
**Причина — отсутствие самой связки**, а не ошибка в логике.

## 2. Глоссарий

| Термин | Значение |
| --- | --- |
| Menu | Набор рецептов на период (например, неделя) с количеством порций. |
| MenuRecipe | Привязка рецепта к меню с переопределением `servings`. |
| ShoppingList | Список покупок, сгенерированный из меню (или ручной). |
| ShoppingListItem | Позиция списка: продукт, количество, единица, прогресс покупки. |
| InventoryItem | Запас продукта в конкретной локации. Количество в base unit. |
| InventoryTxn | Аудит-лог изменений инвентаря (purchase / consumption / adjustment). |
| InventoryLocation | `PANTRY` / `FRIDGE` / `FREEZER`. |

## 3. Функциональные требования

- **FR-2.1** `Menu` имеет имя и опциональные `startDate`/`endDate`.
- **FR-2.2** В меню можно добавлять рецепты с собственным `servings`.
- **FR-2.3** Из меню генерируется `ShoppingList`:
  - все ингредиенты всех рецептов меню,
  - количество масштабируется: `mr.servings / recipe.servings`,
  - агрегируются по `productId`, всё приводится к `Product.baseUnit`,
  - по умолчанию вычитается то, что уже есть в `InventoryItem` (флаг `subtractInventory`, default `true`),
  - позиции с нулём после вычитания не создаются.
- **FR-2.4** Конвертации единиц используют ту же таблицу `UnitConversion`
  из Фазы 1: сначала продукт-специфичная, затем глобальная.
  Если пути нет — генерация падает с понятной ошибкой и НЕ создаёт частичный список.
- **FR-2.5** `PATCH /shopping-list/:listId/items/:itemId/purchase` принимает
  опциональные `quantity` и `location`:
  - `quantity` по умолчанию — оставшееся (`quantity - purchasedQuantity`).
  - **В одной транзакции**:
    1. инкрементируем `ShoppingListItem.purchasedQuantity`;
    2. если `purchasedQuantity >= quantity` — ставим `purchasedAt = now()`;
    3. конвертируем покупаемое количество в `Product.baseUnit`;
    4. апсертим `InventoryItem(productId, location)` — `quantity` инкрементируется;
    5. пишем `InventoryTxn(source=PURCHASE, refType=ShoppingListItem, refId=itemId)`.
- **FR-2.6** `Inventory` API:
  - `GET /inventory` — все позиции, опц. фильтры по `location`, `productId`;
  - `POST /inventory/adjust` — ручная корректировка (+/-) с обязательным `note`,
    пишет `InventoryTxn(source=ADJUSTMENT)`.
- **FR-2.7** Ингредиент рецепта потребляет инвентарь только когда блюдо
  «приготовлено» — в Фазе 2 этот эндпоинт **не делаем** (отдельная фаза).

## 4. Нефункциональные

- **NFR-2.1** Все мутации, затрагивающие более одной таблицы, идут через
  `prisma.$transaction`. Никаких частичных коммитов.
- **NFR-2.2** Инвентарь хранится в `Product.baseUnit` всегда. UI/клиент сам
  решает в чём показать.
- **NFR-2.3** Все `Decimal` поля количества — `Decimal(18, 4)`.
- **NFR-2.4** Юнит-тесты на чистые функции агрегатора и конвертера обязательны.

## 5. Out of scope

- Frontend.
- Списание из инвентаря при «приготовлении» рецепта (Фаза 3).
- Сроки годности, категории магазинов (Фаза 3).
- Multi-user / роли.

## 6. Критерии приёмки

- [ ] `npm run prisma:migrate` — новая миграция `20260518180000_menu_shopping_inventory` применяется чисто.
- [ ] Создание меню с рецептом «Простой томатный салат» (Фаза 1) и
      генерация списка покупок возвращает 3 позиции (помидор / масло / соль)
      в base unit каждого продукта.
- [ ] Если в инвентаре уже лежит 200 г помидоров, в списке окажется
      `300 - 200 = 100 г` (или 0, если хватает).
- [ ] `PATCH /shopping-list/:listId/items/:itemId/purchase` без `quantity`
      закрывает позицию полностью, инвентарь увеличивается ровно на
      покупаемое количество в base unit.
- [ ] При ошибке апсерта инвентаря весь purchase откатывается — позиция
      остаётся не купленной (атомарность).
- [ ] Юнит-тесты агрегатора и конвертера зелёные (`bun test test/`).
