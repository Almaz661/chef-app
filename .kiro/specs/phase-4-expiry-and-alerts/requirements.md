# Фаза 4 — Сроки годности и предупреждения

## 1. Цель

Дать пользователю отслеживать, **что и когда испортится**, и пользоваться запасами в правильном порядке (старое — раньше).

После Фазы 4 ШефДом понимает:
- у каждого купленного продукта может быть дата «годен до»,
- при готовке **сначала уходит то, что скоро испортится** (FEFO — first-expire-first-out),
- есть отдельный список «вот-вот испортится» с настраиваемым горизонтом
  (по умолчанию 3 дня),
- уже просроченные позиции отдельно помечены как «испорчено».

## 2. Глоссарий

| Термин | Значение |
| --- | --- |
| Batch (партия) | Одна запись `InventoryItem` с собственной датой `expiresAt`. У одного продукта в одной локации может быть несколько партий. |
| FEFO | First-Expire-First-Out — приоритет списания: ближайший срок годности первым. |
| Expired | `expiresAt < now`. Партия осталась в инвентаре, но помечена как испорченная — пользователь решает, выбросить или нет. |
| Expiring soon | `expiresAt <= now + horizon`, где `horizon` по умолчанию 3 дня. |

## 3. Функциональные требования

### Партии в инвентаре

- **FR-4.1.** Снять с `InventoryItem` уникальность `(productId, location)`.
  Один продукт в одной локации может иметь несколько строк с разными
  `expiresAt`.
- **FR-4.2.** Добавить в `InventoryItem` поля:
  - `expiresAt: DateTime?` — необязательная дата «годен до»,
  - `acquiredAt: DateTime` — когда позиция оказалась на полке (по умолчанию `now()`).
- **FR-4.3.** Индекс по `expiresAt` для быстрых выборок «что скоро портится».

### Покупки

- **FR-4.4.** В `MarkPurchasedDto` добавляется опциональный `expiresAt`.
  При отметке позиции купленной в инвентаре создаётся **новая партия**
  с этим сроком (`null`, если не указан).
- **FR-4.5.** Поведение по умолчанию (без `expiresAt`) — как раньше:
  партия без срока. Старая идея «merge при upsert» больше не нужна,
  партии всегда отдельны.

### Готовка (FEFO)

- **FR-4.6.** При вызове `POST /menu-recipes/:id/cook` для каждого
  ингредиента:
  - выбираем партии нужного продукта в выбранных локациях
    (`preferLocation → PANTRY → FRIDGE → FREEZER`, как в Фазе 3),
  - **внутри одной локации сортируем партии**:
    1. сначала уже просроченные (`expiresAt < now`),
    2. потом по `expiresAt asc` (`null` — в самом конце),
    3. при равном `expiresAt` — по `acquiredAt asc`.
  - забираем сколько нужно, переходя к следующей партии при исчерпании.
- **FR-4.7.** Если партии исчерпаны до нуля — соответствующая запись
  `InventoryItem` удаляется. Это держит таблицу маленькой и не плодит
  «нулевые» строки.
- **FR-4.8.** Атомарность как в Фазе 3 — всё в одной `prisma.$transaction`.

### Корректировка вручную

- **FR-4.9.** `POST /inventory/adjust` с положительным `quantity`
  создаёт новую партию (опционально с `expiresAt` из DTO).
- **FR-4.10.** `POST /inventory/adjust` с отрицательным `quantity`
  списывает по FEFO из доступных партий выбранной локации (или из
  любой, если локация не указана). Если запасов не хватает —
  откатываем всё.

### Алерты

- **FR-4.11.** `GET /alerts/expiring?days=N` возвращает партии, у
  которых:
  - `expiresAt < now` (статус `EXPIRED`), либо
  - `now <= expiresAt <= now + N days` (статус `SOON`).
  По умолчанию `N = 3`. Сортировка: сначала просрочка, дальше по
  `expiresAt asc`.
- **FR-4.12.** `GET /alerts/expiring/count?days=N` — то же, но только
  числа: `{ expired: int, soon: int }`. Удобно для бейджа в UI.

## 4. Нефункциональные

- **NFR-4.1.** Все мутации — в `prisma.$transaction`.
- **NFR-4.2.** FEFO-выбор — чистая функция (`pickBatches`),
  тестируется без БД.
- **NFR-4.3.** Сравнение дат — через `now: Date` параметр у чистой
  функции, чтобы тесты были детерминированы.

## 5. Out of scope

- Push/Email уведомления.
- Авто-добавление просроченного в shopping-list.
- Notion-style дашборды.

## 6. Критерии приёмки

- [ ] Покупка с `expiresAt = "2026-05-22"` создаёт партию с этой датой
      и не сливается с существующей партией без срока.
- [ ] Готовка списывает сначала ту партию, у которой `expiresAt`
      ближе всего; нулевые партии удаляются.
- [ ] `GET /alerts/expiring?days=3` возвращает уже просроченные +
